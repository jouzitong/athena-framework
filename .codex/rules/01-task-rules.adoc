= Codex Task 定义规范（Task Definition Specification）
:doctype: book
:toc: left
:toclevels: 3
:sectnums:

[perfre]
== 基本原则

- Codex 只能执行状态为 `READY` 的任务
- Codex 执行完成后，必须将任务状态改为 `DONE`
- Codex 不得修改 DSL 结构，只能修改指定 Block
- 任务编号必须全局唯一，禁止重复
- Codex 你执行的任务来源都来自 codex/tasks/ 目录下的任务清单文件。
- 生成任务执行日志并保存到指定目录
- 执行完成任务后，更新任务状态为 `DONE`
- 支持复杂开发场景下的任务拆分与编排

本文档描述的是 **规范（Specification）**， 而不是某一个具体任务的示例或模板。

== 核心概念定义

=== Task（任务）

Task 表示一个**完整、可交付的开发目标**, 通常对应::
- 一个功能需求
- 一个实体或模块的建设
- 一次重构或缺陷修复

=== 2.2 Sub Task（子任务）

Sub Task 是 Task 内部的 **最小可执行单元**。

Sub Task 的特征：

- 可被独立执行
- 可被 Flow 驱动
- 可生成独立的 Execution Report
- 状态可被裁决（DONE / BLOCKED）

=== 2.3 Task 与 Sub Task 的关系

----
Task（文件级）
├─ Task Meta（身份与目标）
├─ Task Completion Policy（整体完成判定）
└─ Sub Tasks（执行单元）
----

== Task 文件结构规范

.对于一个合格的 Task 文件**必须且不局限** 包含以下结构块：
[source,text]
----
Task File
├─ Task Overview
├─ Task Meta
├─ Task Scope（任务范围）
├─ Risks & Notes（风险与注意事项）
├─ Task Completion Policy
└─ Sub Task Blocks（1 个或多个）
----

=== Task Overview（任务概述）

这里主要是描述任务的背景、整体概括、使用的 flows等关键信息

.关键要素
- **任务拆分方式**: 定义如何将整个开发目标拆分为多个可独立执行的子任务。
- **任务之间的依赖关系**: 明确各个子任务之间的依赖关系，确保任务按正确的顺序执行。
- **任务组的完成与失败判定规则**: 定义任务组何时被视为完成或失败，以及相应的判定条件。

.作用
- **可追踪性**: 通过清晰的任务拆分和依赖关系，确保每个任务的状态和进度可以被追踪。
- **可裁决性**: 提供明确的完成和失败判定规则，使得任务组的状态可以被准确地判断。
- **可演进性**: 任务组的结构允许在开发过程中进行调整和优化，以适应不断变化的需求和技术环境。

.建议字段（可选）
- **背景与动机**: 为什么需要该任务，关联的业务/技术背景是什么。
- **输入与前置**: 需要的资料、配置、权限或上游产出。
- **关键约束**: 时间窗口、性能指标、兼容性要求、不可触碰范围等。
- **Flow 选择**: 说明使用哪些 Flow（若有）及其选择理由。
- **验收方式**: 任务成功的验证路径（测试、评审、日志、数据校验等）。

.写作规范
- 聚焦“为什么做”和“做成什么样”，避免描述具体实现步骤。
- 使用要点式表达，保持可扫描性。
- 与 Task Meta / Completion Policy 形成互补，不重复堆叠。

=== Task Meta（任务元信息）

==== Task Meta

task.id（任务ID）:: 任务ID。全局唯一主键，建议包含领域前缀与序号（如 `TV-TASK-000123`），用于索引、追踪与跨文件引用。
task.name（任务名称）:: 任务名称。用一句话概括任务内容与目标范围，避免包含实现细节或操作步骤。
task.type（任务类型）:: 任务类型。用于分类统计、路由 Flow、选择执行策略。取值见「Task Type 类型规范」。
task.version（任务版本）:: 任务版本。用于记录任务定义的版本演进，默认 `V1.0.0`，出现范围/目标变更时递增。
task.status（任务状态）:: 任务状态。用于控制可执行性与生命周期流转，取值见「状态定义」。

==== Task Objective（任务目标）

task.objective（任务目标）::
用 1～3 句话描述该任务最终要交付的能力或结果。 禁止描述具体实现方式。

=== Task Scope（任务范围）

==== Task Scope

task.scope.modules（范围模块）::
列出与该任务直接相关的模块/包/服务/目录，作为影响范围与测试范围的明确边界。建议用可识别的路径或模块名（如 `packages/order-service`）。

task.scope.excludes（范围排除）::
列出明确不在本次任务范围内的模块或能力，避免扩展范围与误判完成条件（如“不改动结算流程”）。

=== Task Completion Policy（任务完成条件）

.字段说明
- *task.completion.success（完成条件）*: 任务完成的判定条件，必须可验证、可操作、可被检查（例如“所有 Sub Task 为 DONE 且关键测试通过”）。避免使用“基本完成”“大致可用”等不可量化描述。
- *task.completion.failure（失败条件）*: 任务失败/终止的判定条件或触发器，用于及时裁决（例如“核心依赖长期 BLOCKED 超过 7 天”或“无法获得必要权限/数据”）。

.建议原则
- 仅使用结果型条件，不写具体实施步骤。
- 条件数量不宜过多，通常 1～3 条足够。
- 能与日志、测试报告或执行报告进行对应验证。

=== Risks & Notes（风险与注意事项）

用于记录任务级别的关键风险、前置假设与执行注意事项，便于评审与裁决。

.写作规范
- 风险应可识别、可验证、可被跟踪，不写泛泛描述（如“可能有风险”）。
- 明确影响范围与触发条件，必要时标注缓解思路或替代方案。
- 注意事项聚焦执行层面的限制（权限、环境、时间窗口），不替代 Task Objective。

.示例
[source,text]
----
Risks & Notes
- 上游订单接口可能在本周变更，需关注协议变动并准备兼容方案
- 回放环境权限受限，需提前申请以避免阻塞
----

=== Sub Task 定义规范

.对于一个合格的 Sub Task 文件**必须且不局限** 包含以下结构块：
[source,text]
----
Sub Task Blocks
├─ Sub Task Overview
├─ Sub Task Meta
├─ Sub Task Completion Policy
├─ Risks & Notes（风险与注意事项）
└─ Execute Report(执行报告)
----

==== Sub Task Overview（子任务概述）

用于描述该子任务的背景、边界与执行方式提示，便于 Flow 选择与执行裁决。

.建议字段（可选）
- **子任务背景**: 该子任务产生的原因或触发条件。
- **输入与依赖**: 需要的配置、接口、文档、或上游产出。
- **影响范围**: 可能触达的模块/服务/文档范围。
- **执行策略提示**: 推荐使用的 Flow 或执行顺序提示。
- **验收方式**: 子任务完成的验证方式或检查清单。

.写作规范
- 保持“可执行性提示”而非“实现步骤”。
- 信息紧凑、结构清晰，避免长段落。
- 与 Sub Task Objective 保持一致，不新增目标。

==== Sub Task Meta

sub_task.id（子任务ID）:: 子任务ID。在本 Task 内唯一即可，建议包含父任务短ID与序号（如 `TV-TASK-000123-ST-01`），便于日志与依赖引用。
sub_task.name（子任务名称）:: 子任务名称。用简洁动宾结构描述要完成的事项（如“新增订单校验规则”），避免包含过多上下文。
sub_task.type（子任务类型）:: 子任务类型。用于驱动 Flow 的执行策略与产出预期，取值见「Sub Task 类型规范」。
sub_task.status（子任务状态）:: 子任务状态。用于执行控制与裁决，取值见「状态定义」。仅 READY 的子任务允许执行。

==== Sub Task Objective（任务目标）

sub_task.objective（子任务目标）::
用 1～3 句话描述该子任务的可交付结果或验证点，不描述具体实现步骤或文件变更方式。

sub_task.depends_on（依赖子任务）::
列出该子任务的前置依赖（子任务ID）。被依赖的子任务必须处于 DONE 状态，当前子任务才可进入 READY。

==== Sub Task Completion Policy（子任务完成条件）

- *sub_task.completion.success（完成条件）*: 子任务完成的可验证条件（如“新增接口在集成测试中通过”）。
- *sub_task.completion.failure（失败条件）*: 子任务失败/终止条件（如“上游接口协议变更导致目标不成立”）。

.建议原则
- 明确可验证信号：测试、检查清单、日志、评审结论等。
- 与 Sub Task Objective 保持一致，避免另起目标。
- 条件尽量原子化，便于裁决。

==== Risks & Notes（风险与注意事项）

用于记录子任务层面的风险、依赖不确定性与执行约束。

.写作规范
- 以“可执行/可阻塞”为核心，说明触发条件与影响。
- 优先记录会导致 BLOCKED 或返工的因素。
- 与 Sub Task Completion Policy 相区分，不重复完成条件。

.示例
[source,text]
----
Risks & Notes
- 依赖的配置中心可能延迟发布，需准备临时开关策略
- 生产环境灰度窗口仅周三晚间可用
----

==== Execute Report（执行报告）

用于记录子任务执行过程与结果，是裁决子任务状态的重要依据。

.建议字段（可选）
- **执行摘要**: 简述做了什么、产出是什么。
- **关键变更**: 影响的模块/配置/文档范围。
- **验证与证据**: 测试结果、日志、截图、对比数据等。
- **问题与结论**: 遇到的问题、解决方式、当前状态结论。

.写作规范
- 以结果和证据为主，避免流水账。
- 可引用具体测试或检查清单，但不粘贴大量日志。
- 与 Sub Task Completion Policy 对齐，便于裁决 DONE / BLOCKED。

.示例
[source,text]
----
Execute Report
- 执行摘要: 完成风控校验规则接入，新增统一错误码返回
- 关键变更: packages/order-validation, docs/error-codes
- 验证与证据: 单元测试通过，集成用例 12/12 通过
- 问题与结论: 无阻塞问题，建议状态置为 DONE
----

== 常见属性定义规范

=== Task Type 类型规范

- **功能 (FUNC)**: 这种任务类型主要用于实现新的功能或改进现有功能。它通常涉及编写新的代码、修改现有代码以添加新特性或增强现有功能。
- **修复bug (BUGFIX)**: 这种任务类型专注于解决软件中的错误或缺陷。它可能涉及调试问题、定位并修复代码中的错误，以及确保修复后的代码通过所有相关测试。
- **重构 (REFACTOR)**: 重构任务的目的是改善代码结构而不改变其外部行为。这可能包括重写部分代码以提高可读性、性能或可维护性，或者将代码迁移到新的架构或技术栈中。
- **文档 (DOCS)**: 这种任务类型主要用于编写或更新文档。它可能包括编写技术文档、用户手册、API 文档等，以确保项目有清晰的文档支持。

=== Sub Task 类型规范

.首先子任务具体需要产出的任务描述, 当前具体存在的需求有
- *FUNC*: 这种子任务类型主要用于修改代码以实现新的功能或改进现有功能。它通常涉及编写新的代码、修改现有代码以添加新特性或增强现有功能。
- *DOCS*: 这种子任务类型主要用于编写或更新文档。它可能包括编写技术文档、用户手册、API 文档等，以确保项目有清晰的文档支持。
- *TEST*: 这种子任务类型主要用于编写和执行测试用例。它可能涉及单元测试、集成测试、端到端测试等，以确保代码的质量和稳定性。

=== Task 与 Flow 的关系

Flow 仅作用于 Sub Task，不直接作用于 Task。

=== 状态定义

.Task 与 Sub Task 状态：
TODO:: 可以理解为新建状态. 不可以执行该任务
READY:: 表示任务已经准备好可以被执行。所有前置条件都已经满足，Codex 可以开始处理这个任务。
DOING:: 表示任务正在被处理。一旦任务从 READY 状态转换为 DOING，意味着 Codex 已经开始工作于该任务。
DONE:: 表示任务已经完成。当任务达到预期结果或目标后，状态应被设置为 DONE。这表明不再需要进一步操作。
BLOCKED:: 当任务因外部因素（如依赖未解决、资源不可用等）而无法继续时使用。处于 BLOCKED 状态的任务暂时不能前进，直到障碍被清除。

== Codex 行为约束

- Codex 不得修改 Task Objective
- Codex 不得跳过 Sub Task 依赖
- Codex 不得直接修改 Task 状态
- Codex 必须以 Sub Task 为执行与汇报单位

== 总结

本规范是 Task DSL 的基础文档， 所有 Task 实例、模板与自动化流程均应以此为依据。
