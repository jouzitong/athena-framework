== 规范说明

本文档用于约束 Codex 在执行任务时的行为规范，
该规范来源于 `flows/task-execution-flow.adoc`。

- 适用对象：所有基于 Task / Sub Task 的执行流程
- 适用范围：代码实现 / 重构 / 文档生成 / DSL 驱动任务
- 约束强度：必须遵守，不得跳过或弱化

== 进入条件（Entry Conditions）

只有当以下所有条件满足时，Codex 才允许进入执行流程：

- 任务状态 = `READY`
- 项目代码规范（Rules）已加载
- 本次执行的任务文件已明确指定
- 当前工作分支 ≠ `main` / `master`

.进入条件不满足时
[source,text]
----
STOP_AND_REPORT
----

进入条件不满足时，Codex 必须：

- 停止执行
- 输出原因说明
- 不得生成或修改任何代码 / 文档

== 任务执行步骤（Execution Steps）

Codex 必须严格按照以下顺序执行步骤，
不允许跳过或重排。

=== Step 1：任务与 DSL 理解（Analyze）

== 关键决策点（Decision Points）

在以下情况下，Codex 必须立即停止执行：

- 发现 DSL 存在歧义 → `REPORT_AND_STOP`
- 违反代码规范 → `REPORT_AND_STOP`
- 修改文件超出 Scope → `STOP`
- 执行目标不明确 → `REPORT_AND_STOP`

任何 STOP 行为：

- 不得进入后续步骤
- 必须生成说明报告

== 可观测性要求（Observability）

每一次任务执行，必须输出以下信息：

- DSL 理解说明
- 执行计划
- 关键决策说明
- 隐含假设列表
- 风险与不确定点

这些输出用于：

- 优化 DSL
- 优化 Flow
- 评估 Codex 执行质量

== 退出条件（Exit Criteria）

当且仅当满足以下条件时，流程视为成功完成：

- 任务状态更新为 `DONE`
- 未发生未处理的规范违规
- Execution Report 已完整生成

=== 失败退出策略

.exit.on_fail
[source,text]
----
TASK_STATUS = BLOCKED
----

== 流程使用强约束（Mandatory Rules）

- 一次只允许执行一个任务
- 不允许同时修改多个任务文件
- 不允许跳过任何执行步骤
- 未通过校验不得进入完成状态
- Codex 不得声称执行了任何系统命令

== 流程设计原则（给人看的）

- Flow 是 Codex 的“行为边界”，不是建议
- Scope 是安全底线，优先级高于任务描述
- 可观测性优先于执行效率
- 宁可中止，也不要“猜着做”
