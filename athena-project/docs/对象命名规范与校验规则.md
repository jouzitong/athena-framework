# 低代码平台对象命名规范（团队标准）

> 目标：
> - 通过**统一、可推断的对象命名**，明确系统分层、职责边界与依赖方向
> - 降低低代码平台在长期演进中的**理解成本、重构风险和扩展成本**
> - 让“对象名本身”成为**架构约束的一部分**

---

## 一、适用范围

本规范适用于：
- 低代码平台后端（Java / Kotlin 为主）
- 前端模型定义（TypeScript / JSON Schema）
- 平台元数据模型（Schema / Definition / Meta 模型）

不适用于：
- 临时脚本、Demo、PoC

---

## 二、总体设计原则（必须遵守）

### 原则 1：对象名必须体现“层级 + 职责”

> **只看类名，就应该能判断它属于哪一层、做什么用**

❌ 错误示例：
- `UserInfo`
- `OrderData`
- `TaskModel`

✅ 正确示例：
- `UserEntity`
- `CreateOrderCommand`
- `TaskDefinition`

---

### 原则 2：对象只服务于一个生命周期

> **禁止一个对象同时承担：输入 + 业务 + 输出**

---

### 原则 3：命名要表达“变化方向”

- **稳定对象**：命名偏抽象、名词化
- **易变对象**：命名偏场景化、动词化

---

## 三、分层与对象命名规范（核心）

### 3.1 Presentation / Interface 层（接口 & 前端交互）

**职责**：
- 表达接口输入输出
- 不包含业务规则

#### 允许的对象类型

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 请求对象 | `XxxRequest` | `CreatePageRequest` |
| 响应对象 | `XxxResponse` | `PageDetailResponse` |
| 传输对象 | `XxxDTO` | `ComponentDTO` |
| 前端视图 | `XxxVO` | `PageVO` |

❌ 禁止：
- `Entity`
- `Repository`
- `DomainService`

---

### 3.2 Application 层（用例 / 编排层）

**职责**：
- 编排业务流程
- 事务边界
- 调用 Domain 层

#### 允许的对象类型

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 用例服务 | `XxxAppService` | `PageAppService` |
| 写操作意图 | `XxxCommand` | `PublishPageCommand` |
| 查询模型 | `XxxQuery` | `PageListQuery` |
| 上下文 | `XxxContext` | `RenderContext` |

❌ 禁止：
- 直接操作数据库
- 出现 `Impl`

---

### 3.3 Domain 层（核心业务层）

**职责**：
- 业务规则
- 状态一致性
- 长期稳定

#### 允许的对象类型

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 实体 | `Xxx` / `XxxEntity` | `Page` / `ComponentEntity` |
| 值对象 | `XxxValueObject` | `VersionValueObject` |
| 领域服务 | `XxxDomainService` | `PermissionDomainService` |
| 仓储接口 | `XxxRepository` | `PageRepository` |

✅ 规则：
- **不出现技术词**（HTTP / DB / JSON）
- **不依赖 Application / Infrastructure**

---

### 3.4 Infrastructure 层（技术实现层）

**职责**：
- 技术细节
- 外部系统适配

#### 允许的对象类型

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 仓储实现 | `XxxRepositoryImpl` | `PageRepositoryMyBatisImpl` |
| 适配器 | `XxxAdapter` | `PaymentAdapter` |
| 外部调用 | `XxxClient` | `AuthClient` |
| 配置 | `XxxConfiguration` | `MyBatisConfiguration` |

---

## 四、低代码平台专属对象命名规范（重点）

### 4.1 元模型（Meta / Definition）

> 用于**描述系统，而不是运行系统**

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 元实体 | `MetaXxx` | `MetaEntity` |
| 定义模型 | `XxxDefinition` | `PageDefinition` |
| Schema | `XxxSchema` | `TableSchema` |
| 配置模型 | `XxxConfig` | `ComponentConfig` |

❗ 规则：
- **禁止直接参与业务计算**

---

### 4.2 运行时模型（Runtime）

| 类型 | 命名规则 | 示例 |
|----|----|----|
| 执行上下文 | `XxxContext` | `ExecutionContext` |
| 运行实体 | `RuntimeXxx` | `RuntimePage` |
| 渲染模型 | `XxxRenderModel` | `PageRenderModel` |

---

## 六、IDE 层命名校验规则（可落地）

### 6.1 包结构约束（示例）

```
com.xxx.platform
 ├─ interface
 ├─ application
 ├─ domain
 └─ infrastructure
```

#### 规则 1：包名 + 类名双重校验

| 包 | 允许类名后缀 |
|----|----|
| interface | Request / Response / DTO / VO |
| application | Command / Query / AppService / Context |
| domain | Entity / ValueObject / DomainService / Repository |
| infrastructure | Impl / Adapter / Client / Configuration |

---

### 6.2 IDE 静态检查（推荐）

#### IntelliJ IDEA

- 使用 **Inspection → Naming Convention**
- 配合 **ArchUnit / Checkstyle**

示例规则（概念）：
- `domain` 包下禁止出现 `DTO`
- `application` 包下禁止出现 `RepositoryImpl`

---

## 七、代码审查（CR）强制检查清单

### 每个新对象，必须回答以下问题：

1. 这个对象属于哪一层？
2. 是否只承担一个职责？
3. 命名是否体现生命周期？
4. 是否违反依赖方向？
5. 是否与低代码元模型 / 运行时模型混用？

❌ 任一项回答不清楚 → 必须重命名或重构

---

## 八、常见反模式（必须避免）

| 反模式 | 问题 |
|----|----|
| `XXXModel` | 职责不清 |
| `XXXBean` | 无语义 |
| 一个 DTO 到处用 | 生命周期混乱 |
| Meta 和 Runtime 混用 | 平台不可维护 |

---

## 九、结语（架构共识）

> **命名不是形式，而是架构契约**

当对象命名清晰时：
- 架构自然清晰
- 新人快速上手
- 重构风险可控
